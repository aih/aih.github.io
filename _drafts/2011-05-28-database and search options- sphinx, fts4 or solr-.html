DB options:
SQLite3 -easy set up, especially in Django
PostgreSQL - large footprint. Takes up too much memory on AWS micro instance.
File system (no database) - Fast, lacks flexibility later.
NoSQL (e.g. MongoDB, CouchDB) - no simple and standard way of indexing from Sphinx or Solr.

Search options:

Sphinx - small, fast, accurate
Solr -fast, flexible, not small
FTS4 (native index and search for SQLite3) - very small, easy import from SQLite3, simple

Problems:
Sphinx - simple indexing of MySQL, PostgreSQL databases. More trouble for files (need to build a script to convert files to an XML pipe); SQLite3 (dump to psql, fix errors, index in Sphinx)

Solr - Powerful options to index files, needs more set-up of the server itself (Java based), more heavyweight--may be too big for AWS micro instance (use Django-haystack + Jetty?)

FTS4 - not many search options (results relevance will suffer)

--

The text format is pure SQL so you can also use the .dump command to export an SQLite database into other popular SQL database engines. Like this:
$ createdb calawdb
$sqlite3 calawdblite .dump | psql calawdb
